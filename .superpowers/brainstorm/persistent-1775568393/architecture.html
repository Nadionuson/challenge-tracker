Architecture Approach

All three use Next.js + TypeScript + Tailwind on Vercel. The difference is how data is persisted.

A

GitHub API as the database Recommended

Data lives in data/challenge.json in the repo. Every check-in writes via the GitHub Contents API — which creates an actual git commit. You get full history, zero extra services, and the JSON is always readable on GitHub. Needs a GitHub Personal Access Token stored as a Vercel env var.

Pros

  • Zero extra accounts or services
  • Full data history in git
  • JSON readable directly on GitHub
  • Free forever

Cons

  • Needs a GitHub PAT (one-time setup)
  • Slight latency on save (~1s API call)
  • Rate limit: 5000 req/day (not a concern here)
B

Vercel KV (Redis) for storage

Data stored in Vercel's built-in Redis KV store. Instant reads/writes, no GitHub API involved. The JSON file in the repo becomes just the initial seed. Requires enabling Vercel KV on the project dashboard (free tier, ~$0/mo for this usage).

Pros

  • Fastest reads/writes
  • Native Vercel integration
  • No GitHub API or PAT needed

Cons

  • Extra service to set up
  • No data history or audit trail
  • Data lives outside the repo
C

Vercel Blob for JSON file storage

Data stored as a JSON blob in Vercel's file storage. Simpler than KV — it's literally a JSON file served from a URL. Easy to inspect, download, and back up. Also requires enabling Vercel Blob on the dashboard (free tier: 1GB).

Pros

  • Closest to "JSON file" mental model
  • Easily downloadable/inspectable
  • Simple API

Cons

  • Extra service to set up
  • No history
  • Less familiar than KV or GitHub API

Common to all three

Next.js 14 App Router · TypeScript · Tailwind · Vercel deployment · Password auth via Next.js middleware + env var · Challenge config in data/challenge.json (seed)
